# KAKASHI-08 Audit Data and Operation Contract

## Objective

Add durable financial history that answers who changed a record, what changed, when,
why, and as part of which application operation. The history must survive destruction
of the business record, remain queryable without loading live associations, and give a
later rollback service enough information to build a guarded compensating operation.

KAKASHI-08 records history. Existing exchange, reference, and Piggy Bank audit services
continue to inspect current graph integrity and are not replaced by this feature.

## Locked Product Decisions

- Use PaperTrail 17 with a custom version class. PaperTrail 17 supports Active Record
  7.1 through 8.1 and Ruby 3.2 or newer.
- Auditing begins when this feature is deployed. Do not synthesize versions or create
  events for existing records.
- Retain financial audit operations and versions indefinitely. There is no deletion UI
  or routine retention purge in this release.
- Administrators can read audit history for every user and context.
- Non-admin users can read only versions owned by themselves.
- Rollback remains admin-only. Read access does not imply correction authority.
- A rollback targets a complete operation, never one record extracted from a cascading
  operation.

Primary library reference:

- [PaperTrail compatibility and documentation](https://github.com/paper-trail-gem/paper_trail)

## Persistence Model

### `AuditOperation`

One immutable operation row describes the application action that caused one or more
versions. Use a UUID primary key so the identifier can be allocated before the first
audited callback runs.

Required columns:

| Column | Type | Contract |
| --- | --- | --- |
| `id` | `uuid` | Stable operation identifier generated by the application |
| `actor_id` | `bigint`, nullable | User that initiated the operation; null for system or unknown work |
| `context_id` | `bigint`, nullable | Context selected at the operation boundary, not necessarily every affected record context |
| `request_id` | `string`, nullable | Rails request ID when the operation originates from HTTP |
| `source` | `string` | Root source such as `web`, `import`, `background_job`, or `rollback` |
| `result` | `string` | `committed`, `rejected`, or `failed` |
| `parent_operation_id` | `uuid`, nullable | Causal parent for later jobs, received actionable messages, or related work |
| `rollback_of_operation_id` | `uuid`, nullable | Original operation compensated by this operation |
| `selected_version_id` | `bigint`, nullable | Version used as the requested rollback boundary, when applicable |
| `metadata` | `jsonb` | Small source-specific identifiers and result details |
| `created_at` | `datetime` | Immutable operation timestamp |

Do not add `updated_at`. Operation rows are written once with their final known result.
Normal operations use `committed`; a failed business transaction rolls its operation
row back with it. A rejected or failed rollback attempt writes a separate immutable
operation row after the business transaction has been abandoned.

### `AuditVersion`

Use a custom PaperTrail version class and an `audit_versions` table. Required
PaperTrail columns are supplemented with authorization and operation metadata.

| Column | Type | Contract |
| --- | --- | --- |
| `id` | `bigint` | Version identifier |
| `item_type` | `string` | Base Active Record type used by PaperTrail |
| `item_subtype` | `string`, nullable | STI subtype, required for `CashInstallment` and `CardInstallment` reification |
| `item_id` | `bigint` | Business-record ID, retained after destruction |
| `event` | `string` | `create`, `update`, or `destroy` |
| `whodunnit` | `string`, nullable | PaperTrail-compatible actor identifier |
| `object` | `jsonb`, nullable | Complete pre-change representation used for reification |
| `object_changes` | `jsonb`, nullable | Before/after attribute changeset |
| `operation_id` | `uuid` | Groups every version produced by one operation |
| `owner_id` | `bigint` | Denormalized owning user used for authorization after destruction |
| `context_id` | `bigint`, nullable | Denormalized record context when the record is context-bound |
| `mutation_source` | `string` | Immediate mutation source, including nested projections or synchronization |
| `metadata` | `jsonb` | Small record-specific routing metadata |
| `created_at` | `datetime` | Version timestamp |

`actor_id`, `owner_id`, and `context_id` are historical identifiers. Do not attach
cascading foreign keys to users or contexts. Deleting or deactivating a live owner must
not erase, mutate, or block historical evidence. `operation_id` may use a restrictive
foreign key because both audit tables are immutable and retained together.

### Indexes

Add indexes for the supported filters instead of beginning with broad JSON indexes:

- `audit_versions(operation_id, id)`
- `audit_versions(item_type, item_id, created_at)`
- `audit_versions(owner_id, created_at)`
- `audit_versions(context_id, created_at)`
- `audit_versions(mutation_source, created_at)`
- `audit_versions(event, created_at)`
- `audit_operations(actor_id, created_at)`
- `audit_operations(context_id, created_at)`
- `audit_operations(source, created_at)`
- `audit_operations(parent_operation_id)`
- `audit_operations(rollback_of_operation_id)`
- `audit_operations(request_id)` where the request ID is present

Add a GIN index to `object_changes` only after a real query requires attribute-level
search. The first release filters by indexed metadata and renders JSON after selection.

## Immutability

- `AuditOperation` and `AuditVersion` reject update and destroy through Active Record.
- PostgreSQL triggers reject `UPDATE` and `DELETE` on both audit tables.
- PaperTrail version limits remain disabled so the library never prunes older rows.
- Audit models are not themselves versioned.
- Normal application code has no repair, edit, or delete action for audit rows.
- Test cleanup may truncate audit tables; production application roles may only insert
  and select.

The indefinite-retention decision removes the need for a privileged purge path. A
future retention change requires a separate migration and operational decision rather
than silently weakening this contract.

## Operation Context

Introduce one request/job-safe context object backed by `ActiveSupport::CurrentAttributes`
and exposed through a small service boundary, for example:

```ruby
Audit::Operation.run(
  actor: current_user,
  context: current_context,
  source: :web,
  request_id: request.request_id
) do
  # Existing application action
end
```

The operation boundary owns:

- operation ID
- actor ID
- selected context ID
- request ID
- root source
- immediate mutation source
- optional parent operation ID
- rollback source identifiers
- bounded metadata

Nested synchronous work reuses the operation ID. Services and callbacks may refine the
immediate mutation source with `Audit::Operation.with_mutation_source(:projection_sync)`
without replacing the root source.

Every mutating controller action receives a default `web` operation. Explicit service
boundaries override it where needed. Console or unwrapped code receives `unknown`, not
a fabricated user. The operation is created lazily when the first version is written,
so GET requests and no-op writes do not create empty history rows.

## Source Vocabulary

Use string-backed, validated values:

Root sources:

- `web`
- `api`
- `actionable_message`
- `admin_repair`
- `import`
- `background_job`
- `rollback`
- `console`
- `unknown`

Immediate mutation sources:

- every root source above
- `shared_sync`
- `projection_sync`
- `reference_sync`
- `piggy_bank_sync`
- `balance_recalculation`

Do not infer a precise source from a callback stack. Callers that need a specialized
source must establish it explicitly around the mutation.

## Asynchronous and Cross-User Work

- Synchronous projections and shared graph updates reuse the initiating operation ID.
- A background job receives the initiating operation as `parent_operation_id` and
  creates a new operation ID because it commits in a separate transaction and time.
- Applying an actionable message creates a new operation owned by the receiving user
  and links the sender-side operation as its parent when known.
- New actionable messages persist the nullable sender-side `audit_operation_id` used
  for that causal link. Existing messages are not backfilled.
- One operation may contain versions with different `owner_id` and `context_id` values.
  This is expected for shared-user financial graphs.
- Non-admin readers see only versions where `owner_id == current_user.id`; they do not
  gain access to the other side of a cross-user operation.
- Admin readers may inspect the complete operation across all owners and contexts.

## Ownership Resolution

Version authorization cannot depend on live associations. Resolve and persist ownership
before the business mutation:

- cash/card transactions, budgets, subscriptions, and investments use direct `user_id`
  and `context_id`
- installments inherit owner/context from their cash or card transaction
- category/entity allocations inherit owner/context from their polymorphic transactable
- exchanges inherit owner/context through `EntityTransaction#transactable`
- references inherit user from `user_card` and use their direct context
- user cards and bank accounts use their direct user and have no single record context
- Piggy Banks inherit owner/context from the source transaction, falling back to the
  return transaction during destruction or historical reconstruction

If ownership cannot be resolved, the audited mutation fails closed. `owner_id` is not
allowed to be null for financial versions.

Add models incrementally by slice. A model is not considered covered until its direct

## Callback-Bypassing Writes

PaperTrail cannot observe `update_column`, `update_columns`, `update_all`, `delete_all`,
`insert_all`, or SQL executed outside Active Record lifecycle callbacks. KAKASHI-08
must inventory every such path touching the initial model scope.

Classify each bypass:

1. Replace with validated model writes when the affected row count is bounded.
2. Use an explicit `Audit::BulkMutation` wrapper that captures before/after snapshots
   and inserts equivalent versions in the same transaction when bulk SQL is necessary.
3. Exclude derived cache fields from version payloads and recalculate them instead of
   auditing mechanical counter writes.
4. Reject an audited bulk path when complete ownership or before-state capture cannot
   be established.

High-risk paths include exchange projection rebuilding, reference/card date
resynchronization, context purge, imports, shared paid-state synchronization, and
balance/count recalculation. `delete_all` paths require explicit pre-destruction
snapshots; otherwise destroyed rows would be permanently undiscoverable.

## Payload and Redaction Policy

- Store `object` and `object_changes` as PostgreSQL `jsonb`, never YAML.
- Skip `created_at` and `updated_at` from changesets.
- Skip derived cache-only totals, counts, and balances when the value is recreated by
  canonical recalculation and is not itself a user decision.
- Keep financial descriptions, comments, prices, dates, paid state, routing IDs,
  allocation values, reference dates, and starting prices.
- Do not add `User`, authentication, session, encrypted credential, token, or secret
  models to the audited scope.
- Metadata accepts only allowlisted scalar identifiers. Do not serialize request
  parameters, headers, cookies, exception backtraces, or arbitrary model objects.
- Enforce bounded metadata and payload sizes. Oversized financial versions fail the
  business mutation visibly rather than silently truncating audit evidence.

## Failure Semantics

- Configure PaperTrail version errors to raise.
- An audit insert failure rolls back the business transaction.
- A business validation failure creates no version.
- A transaction rollback removes every version and the lazily-created operation from
  that transaction.
- Audit callback recursion is prohibited; inserting audit rows never audits audit rows.
- A no-op save creates neither a version nor an operation.

## History Read Surface

Provide shared history queries and Phlex views rather than coupling history to the
current settings audits:

- `/audit_operations` lists operations visible to the current reader
- `/audit_operations/:id` shows one operation and its ordered versions
- a reusable record timeline lists versions for an `item_type` and `item_id`
- transaction and other supported show pages link to the record timeline
- destroyed records remain searchable by stored type, ID, owner, context, operation,
  event, source, actor, request ID, and date range
- human-readable changes use model translations and formatted values
- raw JSON is available in a secondary disclosure, not the default presentation

Authorization is applied in the query object before pagination. Controllers never load
an unrestricted operation and filter its versions afterward.
